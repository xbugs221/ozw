# 首页待处理会话卡片规整

## 用户可感知场景

首页待处理卡片信息分散、Provider 使用文字占位，卡片较窄且会随后台回复刷新、重排。

## 调用链与模块责任

`Provider 会话索引 → 待处理 SQLite 读模型 → 认证接口 → 首页待处理看板`

## 关键证据

- 看板曾监听每个 `session_changed` 消息并重新请求列表。
- SQLite 查询曾按 `last_activity` 排序。
- 卡片曾同时显示 Provider 文案、项目名、会话编号和“处理完成”文字。

## 根因与置信度

Confirmed：活动时间同时承担排序依据和前端失效刷新信号，导致后台回复改变卡片顺序；卡片元信息缺少统一底栏。

## 修复方案

- 列表按创建时间稳定排序，Provider 和会话编号作为同时间的稳定次序后备。
- 前端不再因 Provider 回复事件刷新看板。
- 标题固定两行；完成勾选框、Provider 矢量标志和项目名统一到底栏。
- 标题按内容自适应为一至两行，移除强制两行高度；长标题用满两行后再截断。
- 移除 Provider 标题进入索引前的 20/50/80 字符及 Hermes 256 字节裁剪；仅紧凑路由标签保留 20 字符限制。
- 重新索引时识别旧版五十字符自动摘要并恢复 Provider 完整标题。
- 勾选框直接处理单张卡片，顶部按钮直接处理当前全部卡片，不保留选择状态。
- 看板使用主页全部可用宽度。
