# 规格：测试集分类和中文导读

> 合并自归档提案：`2026-06-05-74-测试集重构中文批注`、`2026-06-06-81-增量引入Vitest测试框架`、`2026-06-06-83-迁移低状态Node测试到Vitest`、`2026-06-06-84-精简Node测试入口并去除重复构建`、`2026-06-06-85-分层Playwright浏览器回归`、`2026-06-06-86-建立分层质量门和耗时基线`

## 需求：测试分类导读必须让非专业审阅者看懂

测试集必须提供中文导读，让不熟悉代码实现的人也能理解每类测试覆盖的业务风险、运行入口和失败含义。

### 场景：根测试说明给出阅读顺序

- **给定** 非专业审阅者第一次打开 `tests/`
- **当** 他阅读 `tests/README.md`
- **则** 文档必须说明建议先读哪些分类导读
- **且** 必须列出 `tests/backend/README.md`、`tests/unit/README.md`、`tests/spec/README.md`、`tests/e2e/README.md`、`tests/manual/README.md`
- **且** 必须说明测试覆盖真实业务需求，不应只做组件冒烟检查

### 场景：每个分类目录都有中文导读

- **给定** 审阅者进入任一测试分类目录
- **当** 他查看该目录 README
- **则** README 必须说明业务场景、运行命令、失败含义和新增测试放置条件
- **且** README 必须使用中文描述，不应只列技术命令

## 需求：关键业务测试源码必须有中文批注

关键测试文件必须在源码中补充中文批注，解释测试背后的业务目的，而不是只让读者从代码推断。

### 场景：代表性测试文件解释业务目的

- **给定** 审阅者打开后端、规格、端到端或手动历史回归中的代表性测试
- **当** 他阅读文件头和关键断言附近批注
- **则** 每个文件必须有中文批注说明该测试保护的业务场景
- **且** 批注必须说明失败时代表的用户风险
- **且** 批注不得只是重复函数名或技术实现细节

### 场景：测试整理不弱化真实回归

- **给定** 执行者补充导读和批注
- **当** 他更新测试目录或测试说明
- **则** 默认测试入口仍必须覆盖 `test:vitest`、`test:server`、`test:spec`、`test:e2e`
- **且** 已分类目录下的真实测试文件不能被移动回 `tests/` 根目录
- **且** 手动历史回归必须继续标明按需运行，不得伪装成默认 CI 已覆盖

## 需求：Vitest 必须作为增量快速测试入口接入

ozw 应新增 Vitest 快速测试入口，用于稳定、无浏览器、少全局状态的业务逻辑测试。该入口必须可被默认回归或明确脚本运行发现，且不能替代现有类型检查、服务端构建、Node 测试和 Playwright 测试。

### 场景：开发者可以运行快速 Vitest 回归

- **给定** 开发者修改了前端或共享层的纯业务逻辑
- **当** 他执行 `pnpm run test:vitest`
- **则** Vitest 必须运行 `tests/unit/**/*.test.ts`
- **且** 至少有 5 个真实业务逻辑测试文件被执行
- **且** 这些测试必须使用真实源码模块和真实业务样例
- **且** 测试覆盖至少两个源码根的真实模块

### 场景：默认测试链路保留既有安全网

- **给定** 维护者阅读 `package.json`
- **当** 他查看 `test`、`typecheck`、`build:server`、`test:server`、`test:spec` 和 `test:e2e`
- **则** `typecheck` 仍必须使用 `tsc -b --noEmit`
- **且** `build:server` 仍必须使用 `tsc -p tsconfig.build.json`
- **且** 服务端和 Node 规格测试仍必须由 `tsx --test` 覆盖
- **且** 浏览器规格和端到端业务流仍必须由 Playwright 覆盖

### 场景：Vitest 配置不会吞掉高状态测试

- **给定** 仓库中存在会改写 HOME、XDG、PATH、WebSocket 或真实浏览器状态的测试
- **当** Vitest 读取配置
- **则** include 必须限制在 `tests/unit/**/*.test.ts`
- **且** 不得匹配 `tests/backend/**/*.test.ts`、`tests/spec/**/*.spec.ts`、`tests/e2e/**/*.spec.ts` 或 `tests/manual/**`

### 场景：低状态迁移不削弱服务端回归

- **给定** 某个测试验证真实服务端读写、临时 HOME、Provider 运行态或 WebSocket
- **当** 维护者把低状态 Node 测试迁入 `tests/unit`
- **则** 高状态测试必须继续由 `pnpm run test:server`、Node 规格测试或 Playwright 入口覆盖
- **且** `test:server` 不得被删除或替换为 Vitest

### 场景：tests/unit 不包含高状态运行态测试

- **给定** 维护者扫描 `tests/unit/**/*.test.ts`
- **当** 检查测试源码中的运行态依赖
- **则** 测试不得启动端口、WebSocket、数据库或长生命周期进程
- **且** 不得依赖真实浏览器或 Playwright

### 场景：测试分类说明指导后续新增测试

- **给定** 后续开发者要新增测试
- **当** 他阅读 `tests/README.md` 和 `tests/unit/README.md`
- **则** 文档必须说明 `tests/unit` 的业务场景、运行命令、失败含义和新增测试规则
- **且** 必须说明 Vitest 适合快速业务逻辑测试，不适合真实浏览器、后端运行态和长链路手动回归

### 场景：分类合同同步接受新的 unit 分类

- **给定** 执行阶段新增 `tests/unit/**/*.test.ts`
- **当** `tests/spec/test_suite_taxonomy.ts` 运行
- **则** 它不得因为 `tests/unit` 存在可执行测试而失败
- **且** 它必须继续阻止可执行测试直接散落在 `tests/` 根目录

## 需求：Node 测试入口必须区分快速 smoke、完整 Node 回归和发布构建

ozw 应把 Node 测试入口拆清楚，避免规格测试重复执行服务端构建，同时保留完整 Node 回归和发布构建验证能力。

### 场景：Node 规格测试不默认触发服务端构建

- **给定** 开发者修改了 Node 规格覆盖的源码逻辑
- **当** 他运行 `pnpm run test:spec:node`
- **则** 命令必须直接运行对应 TypeScript 规格测试
- **且** 不得先运行 `pnpm run build:server`

### 场景：服务端 smoke 子集有明确入口

- **给定** 开发者需要快速确认关键服务端读模型和会话链路
- **当** 他运行 `pnpm run test:server:smoke`
- **则** 应运行明确列出的关键后端测试文件
- **且** 该入口不得伪装成完整服务端回归

### 场景：完整 Node 回归仍可一键运行

- **给定** 开发者不需要打开浏览器，但需要跑完整 Node 回归
- **当** 他运行 `pnpm run test:node`
- **则** 应串联 `test:server` 和 `test:spec:node`
- **且** `test:server` 仍应覆盖 `tests/backend/*.test.ts`

## 需求：Playwright 浏览器回归必须区分快速 smoke 和完整真实业务流

ozw 应提供可快速运行的 Playwright smoke 入口，同时保留完整 browser spec 和 e2e 回归，避免为了速度删除真实用户流程。

### 场景：开发者快速确认关键页面链路

- **给定** 开发者修改了前端或后端轻量逻辑
- **当** 他运行 `pnpm run test:e2e:smoke`
- **则** Playwright 应启动真实 fixture 和应用
- **且** 只运行少量关键 e2e 业务流

### 场景：完整浏览器回归仍可运行

- **给定** 合并前需要确认完整浏览器行为
- **当** 开发者运行 `pnpm run test:browser:full`
- **则** 应串联 browser spec 和 e2e full
- **且** 不应包含 `tests/manual/browser-history`

### 场景：诊断证据只在失败时保留

- **给定** Playwright 测试通过
- **当** 查看测试产物
- **则** 不应因为默认配置生成大量 trace、video 或 screenshot
- **但** 失败时应保留足够证据用于复查

## 需求：测试和编译必须有分层质量门和可复查耗时数据

ozw 应提供面向不同风险级别的测试入口，并能采集关键命令耗时，帮助开发者在效率和覆盖之间做明确选择。

### 场景：开发者运行本地快速质量门

- **给定** 开发者做了小范围逻辑改动
- **当** 他运行 `pnpm run test:fast`
- **则** 应运行无浏览器的快速检查
- **且** 不应运行完整 Playwright e2e

### 场景：开发者运行提交前 smoke

- **给定** 改动可能影响页面入口或会话基础链路
- **当** 他运行 `pnpm run test:smoke`
- **则** 应先运行 fast
- **且** 应运行 e2e smoke

### 场景：合并前运行完整回归

- **给定** 改动准备合并
- **当** 开发者运行 `pnpm run test:full` 或 `pnpm test`
- **则** 应覆盖 typecheck、Vitest、完整 Node 和完整 browser 回归

### 场景：耗时基线可复查

- **给定** 开发者运行耗时采集脚本
- **当** 命令完成
- **则** 应输出包含命令、耗时和退出码的 JSON
- **且** 失败命令不应被记录成成功
