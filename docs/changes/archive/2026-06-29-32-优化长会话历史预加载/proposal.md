# 提案：优化长会话历史预加载

## 背景

现有实现已经具备“最新分页 + 向上翻页 + 虚拟渲染”的基础能力。问题不在于没有分页接口，而在于前端触发更早历史加载的时机过晚。

| 模块 | 当前观察 | 用户影响 |
| --- | --- | --- |
| 前端滚动控制 | 历史加载由接近顶部的滚动条件触发 | 用户必须接近或到达顶部才看到加载动作 |
| 后端消息接口 | 支持 `limit`、`offset`、`nextRawLineOffset` 和 `hasMore` | 可以继续按页读取更早 JSONL 历史 |
| 现有测试 | 端到端测试覆盖 `scrollTop = 0` 后能加载 | 没覆盖“到顶部之前就应预加载”的体验 |
| 虚拟列表 | DOM 渲染窗口有上限 | 可以增加已加载数据量，不必同步渲染全部 DOM |

## 要做什么

1. 在前端会话滚动控制中引入“历史预加载区”判断。
2. 用户向上滚动到预加载区时，后台请求更早消息页。
3. 加载完成后继续使用现有滚动锚点恢复，避免阅读位置跳动。
4. 保留分页请求和 DOM 虚拟化上限，禁止默认全量拉取。
5. 增加合同测试和根目录回归测试，覆盖“未到顶部先发请求”的行为。

## 为什么这样做

| 取舍 | 结论 |
| --- | --- |
| 只扩大初始加载数量 | 不采用；会拉长进入会话的首屏耗时 |
| 到顶部再加载 | 不采用；这正是当前用户体感问题 |
| 默认加载全部历史 | 不采用；长会话会破坏性能目标 |
| 预加载区内按页后台加载 | 采用；保留性能边界，同时改善翻阅连续性 |

## 成功标准

- 初始进入长会话仍只请求最新分页。
- 在 `scrollTop > 100` 的预加载位置，已经能观察到更早分页请求。
- 更早页 prepend 后，滚动位置仍在用户原阅读区附近。
- 消息 DOM 数量继续受虚拟列表上限保护。
- 既有顶部加载、外部追加、afterLine 增量读取测试继续通过。
