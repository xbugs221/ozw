# 设计：长会话历史预加载

## 根因分析

当前链路可以拆成：

| 环节 | 责任 | 当前问题 |
| --- | --- | --- |
| 前端会话运行时 | 维护消息页、游标、滚动锚点和加载状态 | 更早历史的触发条件过于靠近顶部 |
| 前端消息面板 | 虚拟渲染当前可见窗口 | 不是根因，但必须继续保持 DOM 上限 |
| 后端 transcript 读模型 | 根据 raw line cursor 返回分页消息 | 已具备按页读取能力，暂不作为主责修复模块 |
| Playwright 夹具 | 提供真实长会话浏览器流程 | 现有测试只证明到顶后能加载，没有证明提前预加载 |

## 技术决策

| 决策 | 做法 | 风险控制 |
| --- | --- | --- |
| 预加载触发 | 用视口相关的上方剩余距离判断，而不是固定“顶部 100px” | 合同测试只要求 `scrollTop > 100` 时能提前请求，避免锁死具体算法 |
| 分页策略 | 继续调用现有 `limit=100` 历史页请求 | 验收禁止未显式操作时出现无界请求 |
| 并发控制 | 继续保留加载中保护，避免同一滚动区重复打爆请求 | 执行阶段需检查快速滚动和连续 scroll 事件 |
| 滚动恢复 | 复用现有 prepend 前后 `scrollHeight` 差值锚点 | 合同测试断言加载后 `scrollTop` 仍大于 100 |
| 虚拟渲染 | 数据可按页增加，DOM 仍受窗口上限控制 | 性能边界测试和合同测试都检查 DOM 数量 |

## 接口契约

| 方向 | 契约 |
| --- | --- |
| 前端到后端 | 请求仍使用 `/messages?limit=100&offset=<rawLineCursor>` 或现有等价项目会话路由 |
| 后端到前端 | 响应继续返回 `messages`、`total`、`hasMore`、`nextRawLineOffset` |
| 前端内部 | `messagesOffsetRef` 继续以后端 raw line cursor 推进，不能用 UI message 数量替代 |
| 用户可见 | 未到顶部时后台预加载；到达更早区域时历史内容已经可用或正在无卡顿补齐 |

## 风险

| 风险 | 处理 |
| --- | --- |
| 预加载阈值过大导致多拉页 | 只按页加载，保持 in-flight 保护，合同测试禁止无界请求 |
| 阈值过小仍有卡顿 | 合同测试要求在旧 100px 顶部阈值之外就发起请求 |
| prepend 后阅读位置跳动 | 继续使用滚动锚点，合同测试和既有锚点测试共同覆盖 |
| 与外部追加最新消息冲突 | 保留 `external append while scrolled up` 既有端到端回归 |
| Codex raw line 与 UI message 数量不一致 | 保留 `session-incremental-read` 与 Codex message order 回归 |

## 执行建议

执行阶段优先把预加载判断抽成小型纯函数，便于单测锁定阈值边界；再在 `sessionRuntimeController` 中替换当前接近顶部的硬阈值。若实现证明需要微调后端分页语义，必须同步更新本提案的 `design.md`、`spec.md` 和 `acceptance.json`。
