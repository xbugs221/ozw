# 文件目的：说明终端统一入口与 tmux 保活提案的执行上下文。

## 用户问题

当前会话入口和终端入口的产品语义容易分裂：点击会话卡片时，用户真正想做的是回到一个可操作终端继续运行 `codex` 或 `pi`，而不是默认进入 JSONL 富渲染记录页。同时，如果浏览器刷新或网络断开，基于 WebSocket 生命周期的终端进程可能丢失运行现场。

## 交付目标

| 目标 | 说明 |
| --- | --- |
| 终端统一 | 不区分“普通终端”和“会话终端”，所有终端都是普通 shell |
| 会话入口命令化 | 点击会话卡片或新建会话时，只是打开终端并注入 `codex resume` / `pi --session` / 新建命令 |
| 断线保活 | 用 `tmux` 或等价后端常驻层维持终端进程，浏览器断开只 detach |
| 记录显式查看 | JSONL 渲染入口改为“记录/详情”，用户主动点击才加载 |
| 桌面布局简化 | 桌面终端不再固定底部 dock，而是和会话记录/详情等主工作区视图平行替换 |

## 非目标

| 非目标 | 原因 |
| --- | --- |
| 自动识别用户在终端手动启动的新会话 | 容易误绑定 sessionId，破坏用户自由 shell 心智 |
| 把终端状态强绑定到业务 sessionId | 用户 `Ctrl-C` 退出 TUI 后应能继续把终端当普通 shell 用 |
| 提供跨机器远端生产热修能力 | 本变更只处理本地 Web 工作台的终端生命周期 |

## 验收入口

| 类型 | 入口 |
| --- | --- |
| 契约测试 | `docs/changes/37-终端统一入口与tmux保活/tests/*.acceptance.test.ts` |
| 核心 QA | 点击会话卡片、新建会话、浏览器刷新、模拟 WebSocket 断连、`Ctrl-C` 后继续执行普通命令 |
| 证据目录 | `test-results/terminal-tmux-entry/` |

## 执行阶段默认上下文

执行者应优先改造现有 `shell-websocket`、`Shell`、`StandaloneShell`、项目总览会话卡片和桌面布局，而不是另建一套“会话终端”。实现可以在内部维护 `terminalId`、`sessionId`、`tmuxSessionName` 等字段，但界面和用户语义只暴露“终端”和“记录/详情”。
