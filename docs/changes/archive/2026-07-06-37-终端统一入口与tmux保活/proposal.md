# 文件目的：定义终端统一入口与 tmux 保活的变更提案。

## 需求/问题

用户点击会话卡片时，默认意图是继续操作一个终端中的 TUI。现有模型容易把“会话页”“会话终端”“普通终端”“JSONL 渲染页”混在一起，导致入口含义不稳定。更严重的是，如果终端进程跟浏览器 WebSocket 生命周期绑定，网络断连后运行现场会丢失。

## 原因分析

| 现象 | 根因 |
| --- | --- |
| 会话卡片默认渲染 JSONL 详情 | 把历史记录视图当成继续工作的主入口 |
| provider TUI 和 plain shell 行为不同 | 后端用 `isPlainShell` 分支决定是否保活 |
| 终端固定底部 | 桌面主工作区和终端工作区不是平行关系 |
| 断线恢复依赖进程内 PTY 缓存 | 服务重启或 WebSocket 生命周期变化时缺少 `tmux` 级承载 |

## 做法

| 方向 | 做法 |
| --- | --- |
| 统一终端语义 | 所有终端都只是普通 shell；点击会话只注入启动命令 |
| 命令注入 | Codex 恢复用 `codex resume <providerSessionId>` 或当前仓库既定等价命令；Pi 恢复用 `pi --session <providerSessionId>` |
| 新建会话 | 选择 provider 后打开终端并注入新建命令，不先进入 JSONL 详情 |
| `tmux` 保活 | 后端为每个 `terminalId` 创建或复用 `tmux` session，浏览器连接只 attach/detach |
| 记录入口 | 在会话卡片或会话页提供“记录/详情”入口，从 JSONL 加载渲染 |
| 桌面布局 | 终端作为主工作区平行视图，不再默认出现在底部 dock |

## 成果

完成后用户心智应变成：

```text
点击会话卡片
  -> 打开终端
  -> 系统注入恢复命令
  -> 用户在同一个普通 shell 中继续操作

点击记录/详情
  -> 从 jsonl 加载结构化历史
```

断线恢复应变成：

```text
浏览器 / 网络断开
  -> WebSocket 断开
  -> tmux session 继续运行
  -> 浏览器恢复后重新 attach
```
