# 提案：会话消息导航书签

## 为什么要做

长会话里用户经常需要回看之前提出的问题、约束或决策点。仅靠滚动查找效率很低，尤其当会话包含大量工具输出、代码块和多轮回复时，用户很难判断某个历史问题对应的最终回复在哪里。

本次变更增加一个当前会话内的导航书签。书签以用户消息为锚点，并展示对应智能体最终回复正文的前 50 个字符，帮助用户快速识别和跳转。

## 做什么

```text
当前会话消息
  |
  +-- 用户消息 A
  |     +-- 智能体最终回复 A -> 截取前 50 个字符作为摘要
  |
  +-- 用户消息 B
        +-- 智能体最终回复 B -> 截取前 50 个字符作为摘要

消息书签
  |
  +-- 点击 A -> 定位用户消息 A
  +-- 点击 B -> 定位用户消息 B
```

### 行为要求

- 书签只基于当前会话消息。
- 每条用户消息生成一个书签。
- 摘要直接截取后续智能体最终回复正文的前 50 个字符，不调用模型总结，不额外追加省略号。
- 没有最终回复时，书签显示进行中状态。
- 点击书签后滚动到对应用户消息，并让目标消息短暂高亮。
- 如果目标消息不在当前加载窗口内，复用现有逐页加载和定位能力。

### 适配要求

| 设备 | 行为 |
|---|---|
| 桌面端 | 聊天区旁展示常驻书签列表，可滚动，不遮挡输入框 |
| 手机端 | 展示书签按钮，点击后打开抽屉列表，关闭后回到聊天 |

### 性能要求

- 不因为展示书签而自动加载全部历史。
- 不把长会话所有消息都挂载到 DOM。
- 书签索引构建应是线性复杂度，避免每次渲染重复嵌套扫描。
- 继续保持聊天消息虚拟滚动窗口上限。

## 不做什么

- 不做跨会话书签。
- 不做服务端持久化书签。
- 不做全文搜索替代。
- 不处理模型生成摘要或语义摘要。
