# 提案：让 Render 按视口渐进加载历史

## 为什么要改

Render 的首屏优化把“不要一次性渲染完整历史”误实现成“只保留最新固定窗口”。这同时损害了数据可达性和设备适配：最后一个 turn 可能包含大量思考与工具记录，固定 100 条原始记录仍然只形成一个折叠后的展示回合；更早消息虽然存在于 JSONL，却无法通过 Render 上翻访问。

用户真正需要的是一个受控的滑动数据窗口：首次只准备两个屏幕高度，其中一屏供当前阅读、一屏作为上翻预留；只有用户进入预留屏时才继续读取更早历史。数据分页与 DOM 挂载必须分离，加载到内存不等于全部创建前端组件。

## 做什么

1. 让 Render 以实际滚动容器高度和折叠后展示块高度计算页面容量。
2. 用有界小批次填充当前页与一页预留，满足预算后立即停止请求。
3. 把 Render 滚动容器接入独立的历史分页、锁、迟滞和锚点恢复逻辑。
4. 为快照状态增加“前插一页”和分页游标，而不是每次整体替换固定数组。
5. 保留虚拟列表上限，并补齐所有折叠层级的延迟挂载。
6. 修订与真实意图冲突的性能规格和回归测试：禁止自动全量加载，但允许视口驱动的按需分页。

## 预期结果

| 用户动作 | 数据行为 | 页面行为 |
| --- | --- | --- |
| 点击 Render | 只读取足够覆盖约两个视口的尾部数据 | 最新消息立即可见 |
| 停留最新页 | 不再请求更早历史 | 页面稳定，不出现后台扫描 |
| 上翻进入预留页 | 读取更早一个逻辑页并前插 | 当前阅读锚点不跳动 |
| 继续上翻 | 每次进入新的预留页才再次读取 | 最终可按需到达任意旧消息 |
| 不展开折叠项 | 只保留摘要和轻量布局数据 | 重内容节点不存在 |
| 点击折叠项 | 只创建被展开分支的内容 | 其他折叠分支仍不挂载 |
