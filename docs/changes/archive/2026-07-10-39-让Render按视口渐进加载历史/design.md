# 设计：视口驱动的 Render 历史窗口

## 责任边界

| 模块 | 责任 |
| --- | --- |
| 后端消息接口 | 继续按原始 JSONL 游标提供有界分页、总量和 `hasMore`，不感知浏览器视口 |
| Render 快照状态 | 保存已加载尾部窗口、最早游标、是否还有历史、加载锁和快照版本 |
| 视口预算控制器 | 根据容器高度、折叠后展示块高度和滚动位置判断是否需要补页 |
| 虚拟列表 | 只挂载视口附近展示块，已加载但离屏的数据只参与布局索引 |
| 折叠组件 | 关闭时只挂载摘要；打开对应层级后才创建子消息和重内容 |

## 数据流

```text
打开 Render
   ↓
请求一个有上限的尾部小批次
   ↓
折叠分组并测量实际展示高度
   ↓
不足 2 × 视口高度？──是──> 再取一个小批次
   │                              │
   └────────────否<───────────────┘
   ↓
停止请求并定位最新消息
   ↓
用户上翻进入预留屏
   ↓
前插更早批次，恢复滚动锚点，只补回一屏预留
```

## 关键决策

### 1. 页面容量使用像素高度，不使用固定消息数

原始 JSONL 记录会经过过滤、去重、工具调用合并和 turn 折叠，一条展示块可能对应几十条原始记录。因此页面容量定义为滚动容器的 `clientHeight`，加载目标定义为折叠后展示布局至少覆盖两个视口高度。

首个请求仍需有保守的小批次上限；后续根据测量结果继续补批次。单个请求不得无 `limit`，一次填充必须有请求次数保护、会话切换取消和无进展停止条件。

### 2. 只保留一页预取，不后台补全

最新页稳定后，控制器不得继续请求。用户滚动到距离已加载顶部不超过一个视口时，才启动下一次逻辑页填充；填充到当前视口上方重新拥有约一屏缓冲后停止。

使用加载锁和迟滞距离防止同一次滚动产生重复请求。用户回到更靠下区域后才解锁下一次上翻触发。

### 3. 冻结快照允许用户导航引起的前插

“冻结”只表示 Provider 完成事件、项目刷新和外部追加不会自动替换用户正在阅读的内容。用户滚动、搜索或书签定位属于显式导航，可以前插所需历史并增加分页修订号，但不能把实时尾部自动混入当前快照。

快照状态至少包含：

- 已加载并转换后的消息集合；
- 最早原始行游标或 offset；
- `hasMore`、加载状态、请求代次和无进展保护；
- 当前视口预算与已加载布局高度；
- 前插前的锚点键及相对位置。

### 4. 复用锚点算法，隔离滚动容器

Render 使用独立滚动容器，不能继续传入空滚动处理。应复用普通历史的“前插前记录 scrollHeight/scrollTop，前插后恢复差值”能力，但状态和引用必须属于 Render，避免影响隐藏的 TUI 或普通会话状态。

### 5. 数据窗口与 DOM 窗口分离

已加载的两页或用户按需访问过的页面可以保留在内存中，方便向下返回和搜索定位；DOM 仍由虚拟列表限制在视口附近及少量缓冲，展示块上限不得超过现有 150。

### 6. 每一层折叠都必须延迟创建子内容

关闭的 turn 组、工具组、工具结果、diff、大代码块和子任务容器不得仅靠 CSS 隐藏已经创建的内容。每层都要以打开状态作为子树创建条件；摘要所需的计数、标题和最后一行可以在轻量数据阶段计算。

## 视口变化

桌面缩放、窗口调整和手机方向变化后重新测量容量：

- 如果已有数据仍覆盖当前页和一页预留，不请求；
- 如果视口变大导致预留不足，只补到两页预算；
- 如果视口变小，不丢数据、不立即请求，也不扩大 DOM。

## 失败处理与可观测性

- 分页失败时保留已显示内容和阅读位置，不清空快照。
- 下一次上翻或显式重试可以重新请求同一游标。
- Render 面板提供测试可读的加载阶段、`hasMore` 和分页修订信息；浏览器证据记录请求的 `limit`、offset/cursor、触发前后滚动状态和 DOM 节点数。

## 风险与取舍

- 展示块高度只有挂载后才能精确测量，首个小批次可能需要二次补充；允许少量超出两屏，但不得退化成固定大窗口。
- 单个超高但折叠后很短的 turn 可能消耗多个原始分页请求；必须用有界循环和无进展保护避免卡死。
- 搜索命中很旧的消息时可以逐页读取直到目标，这是显式导航，不受“一页预留”限制，但仍不得一次全量读取或全量挂载 DOM。
