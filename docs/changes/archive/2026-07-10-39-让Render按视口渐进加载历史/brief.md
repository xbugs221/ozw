# 39：让 Render 按视口渐进加载历史

## 用户问题

Render 当前把最新固定消息窗口复制为冻结快照，并禁用旧历史分页。长 turn 会占满固定窗口，导致用户只能看到最后一轮；固定 100 条原始记录也无法代表桌面或手机实际能显示的一页。

## 交付目标

- Render 首次出现时只准备“当前可见一页 + 上一页预留”，不扫描完整历史。
- 页面根据实际视口高度和折叠后的展示块高度决定一页容量，不用固定消息条数冒充设备容量。
- 用户停留在最新页时不继续读取旧历史；进入预留页后才按有界小批次读取更早记录，并恢复一页预留。
- 前插更早记录后保持用户正在阅读的位置，避免跳到新页顶部或最新消息。
- 已加载数据可以保留在内存中，但真实 DOM 只挂载视口附近展示块；折叠的思考、工具、diff、大输出和子任务内容在对应展开动作前不得创建重内容节点。

## 非目标

- 不在后台自动补齐完整会话。
- 不把后端原始 JSONL 条数直接定义为“一页消息数”。
- 不改变 Provider JSONL 的数据格式、会话归属或鉴权边界。
- 不让自动完成事件改写用户正在阅读的快照；本提案只允许用户滚动、搜索或书签导航触发受控分页。

## 验收入口

- 真实浏览器入口：`/session/fixture-mixed-long-virtual-session`，点击顶部 Render Tab。
- 真实数据链：隔离 HOME 中的 Codex JSONL → 后端 `/messages` 分页接口 → React Render 快照 → 浏览器滚动与折叠交互。
- 核心合同：`docs/changes/39-让Render按视口渐进加载历史/tests/render-viewport-demand-loading.spec.ts`。

## 执行阶段默认上下文

现有普通聊天历史已经具备顶部预取、分页锁和前插锚点能力；现有虚拟列表已经能测量视口及展示块高度。执行阶段应提取并复用这些能力，让 Render 快照拥有独立的按需分页状态，不得恢复旧的全量分页循环。

## 创建阶段基线

- 两页预算合同按预期失败：桌面快照实际高度约 8231px，按当前视口计算的验收上限约 2293px。
- 上翻分页合同按预期失败：进入预留区后 5 秒内没有更早消息请求。
- 虚拟 DOM 与折叠延迟挂载合同通过，说明这部分现有能力可以保留复用。
